🍱  Pattinathil-Pasi  
Smart  ML-Powered  Excess  Food  Routing  System   
🎯  Core  Idea  (In  Simple  Words)  
When  a  wedding  hall,  hotel,  or  event  has  extra  food ,  instead  of  manually  calling  NGOs…  
Our  system:  
1.  Finds  nearby  NGOs  2.  Uses  Machine  Learning  to  predict:  ○  Who  is  most  likely  to  accept  ○  Who  can  arrive  fastest  3.  Ranks  them  intelligently  4.  Sends  alert  only  to  the  best  ones  5.  Assigns  pickup  automatically  6.  Learns  from  every  outcome  
It  is  not  predicting  waste.  
It
 
is
 
optimizing
 
rescue
 
in
 
real
 
time.
 
 
🧠  What  Problem  Are  We  Solving?  
Current  problem:  
●  Food  providers  don’t  know  which  NGO  is  available  ●  Calls  are  random  ●  Many  NGOs  ignore  requests  ●  Food  expires  while  coordination  happens  ●  No  performance  tracking  
Our  solution:  
A  real-time  intelligent  rescue  routing  engine.

🔥  FULL  END-TO-END  WORKFLOW  
 
🟢  STEP  1—Provider  Creates  Rescue  
Request
 
(Trigger
 
Event)
 
Provider  logs  in  and  submits:  
●  Meals  available  ●  Food  type  ●  Ready  time  -  pickup  time  ●  Expiry  time  -  shelf  life  ●  GPS  location  (auto  captured)  ●  Event  type  ●  Cause  tag  (why  surplus  happened)   
📌  Cause  Tag  (Fairness  Constraint)  
Options:  
●  Overestimated  attendance  ●  Guest  no-show  ●  Weather  issue  ●  Buffer  cooking  policy  ●  Unknown  
This  ensures:  
●  Providers  are  not  unfairly  penalized  ●  Surplus  amount  does  NOT  reduce  score  
Request  becomes:  
LIVE  FOOD  RESCUE  EVENT  
This  event  triggers  the  ML  engine.  
 
 
🟢  STEP  2  —  Nearby  NGOs  Are  Filtered  
Backend  does:  
1.  Reads  provider  GPS  2.  Fetches  NGO  list  3.  Filters  NGOs  within  radius  (e.g.,  15  km)  4.  Calculates  distance  using:  ○  Haversine  formula  OR  ○  Maps  API  
Now  we  have  candidate  NGOs.  
 
🟢  STEP  3  —  ML  Feature  Builder  (Critical  
Part)
 ->Send  to  set  of  ngo’s  filtered  by  distance  +  reliability  score  
For  each  NGO  candidate,  system  creates  a  feature  vector.  
🔍  ML  Feature  Constraints  
Only  use  features  available  before  decision  time .  
We  use  historical  +  contextual  features:  
distance_km  time_of_day  day_of_week  ngo_accept_rate  ngo_avg_response_time  Past_pickups  (NGO)  Recent_activity_count  -  isactive?(0/1)  ngo_reliability_score  =  accept  rate*avg  resp  time*past  pickups   
These  are  safe,  realistic,  and  available  in  real  time.  
This  prevents  data  leakage  —  very  important  in  viva.

🟢  STEP  5  —  Smart  Ranking  Engine  
Now  system  calculates  final  score:  
final_score  =  0.5  ×  acceptance_probability  +  0.3  ×  (1  /  actual  time  collected)  +  0.2  ×  reliability_index  
This  balances:  
●  Likelihood  of  accepting  ●  Speed  ●  Past  reliability
